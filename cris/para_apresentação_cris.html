Roteiro para Apresentação

O que mostrar ao professor, em qual ordem mostrar e como explicar cada parte do sistema Pokedex em Grafos com Docker.

Foco: demonstrar que a aplicação e o ambiente Docker funcionam

1. Ideia principal da apresentação

A apresentação deve mostrar que o sistema foi desenvolvido e preparado para rodar com Docker. O ponto principal não é apenas abrir a tela da Pokedex, mas provar que existe uma aplicação em container, um banco Neo4j em outro container, uma rede Docker personalizada, volumes de persistência e comandos básicos funcionando.

Fala sugerida:

"Nosso projeto é uma Pokedex em Grafos. A aplicação roda em Node.js com Express, o frontend é servido pela própria aplicação, e o banco de dados é o Neo4j. Usamos Docker para subir tudo de forma organizada, com um container para a aplicação e outro para o banco."

2. Primeiro passo: abrir a pasta certa

No terminal, entre na raiz do projeto:

cd "C:\Users\Kaique Alves\Downloads\pokedex-backend\pokedex"

Essa é a pasta correta porque nela estão os arquivos principais do ambiente: Dockerfile, docker-compose.yml, backend e frontend.

3. Mostrar os arquivos principais no VS Code

Arquivo O que falar
Dockerfile Explique que ele é a receita da imagem da aplicação. Ele compila o frontend, instala as dependências do backend e define o comando que inicia o servidor.
docker-compose.yml Explique que ele sobe os serviços automaticamente: app e neo4j. Também configura portas, variáveis de ambiente, rede e volumes.
backend/src/database.js Mostre que o backend se conecta ao Neo4j usando variáveis de ambiente.
backend/src/seed.js Explique que esse arquivo cria os dados iniciais do banco, como Pokémon, tipos, habilidades, usuário admin e relacionamentos.
backend/src/controllers Explique que os controllers guardam a lógica de login, Pokémon, favoritos, equipes e consultas ao grafo.

4. Subir ou confirmar que o sistema está rodando

Se for subir do zero, use:

docker compose up --build

Se o sistema já estiver rodando, abra outro terminal e confirme com:

docker ps

O resultado esperado é aparecerem dois containers:

  • pokedex-app: aplicação Node.js/Express com frontend.
  • pokedex-neo4j: banco de dados Neo4j.
Como explicar:

"O docker ps prova que os containers estão em execução. O container da aplicação publica a porta 3000, e o container do Neo4j publica as portas 7474 e 7687."

5. Mostrar a aplicação no navegador

Abra no navegador:

http://localhost:3000/app

Mostre a tela da Pokedex carregando. Se aparecer "API online", explique que o frontend está conseguindo se comunicar com o backend.

Fala sugerida:

"Essa tela está sendo entregue pelo container pokedex-app. O frontend foi compilado durante a criação da imagem, e o backend Express serve essa interface na porta 3000."

6. Mostrar o banco Neo4j no navegador

Abra:

http://localhost:7474

Use os dados de acesso:

Usuário: neo4j
Senha: pokedex123

Depois execute a consulta:

MATCH (n)-[r]->(m) RETURN n, r, m LIMIT 50;

Essa consulta mostra os nós e relacionamentos do grafo. Os nós representam entidades como Pokemon, Tipo, Usuario, Equipe e Habilidade. As setas representam relacionamentos como FAVORITOU, TEM_TIPO, TEM_MEMBRO e EVOLUI_PARA.

7. Mostrar comandos Docker básicos

Comando O que ele demonstra
docker run --rm hello-world Mostra que o Docker consegue executar uma imagem simples.
docker ps Mostra os containers em execução, incluindo pokedex-app e pokedex-neo4j.
docker images Mostra as imagens existentes, incluindo a imagem personalizada pokedex-app.
docker network ls --filter name=pokedex Mostra a rede personalizada criada para o projeto.
docker volume ls --filter name=pokedex Mostra os volumes usados pelo Neo4j para persistência.

Como demonstrar docker stop e docker rm sem derrubar o sistema

Não pare os containers principais enquanto estiver apresentando, porque isso derruba a aplicação. Para demonstrar stop e rm, use um container de teste: 

docker run --name teste-stop-rm -d node:20-alpine tail -f /dev/null
docker ps --filter name=teste-stop-rm
docker stop teste-stop-rm
docker rm teste-stop-rm
Cuidado:

Evite rodar docker stop pokedex-app ou docker stop pokedex-neo4j durante a demonstração principal, porque isso pode tirar o sistema do ar.

8. Explicar onde os dados ficam armazenados

Os dados da aplicação ficam no Neo4j. Como o Neo4j está dentro de um container, o projeto usa volumes Docker para persistir os dados. O volume principal é o pokedex_neo4j_data, que guarda os arquivos do banco. 

docker volume ls --filter name=pokedex
Fala sugerida:

"O container pode ser recriado, mas os dados importantes ficam no volume. Por isso favoritos, equipes e cadastros podem continuar existindo mesmo após reinicialização do container, desde que o volume seja mantido."

9. Explicar rapidamente a parte do backend

Parte do backend Como explicar
database.js Cria a conexão com o Neo4j usando NEO4J_URI, NEO4J_USER e NEO4J_PASSWORD.
authController.js Cuida do login e cadastro. Usa bcrypt para senha e JWT para autenticação.
pokemonController.js Cuida do cadastro e consulta de Pokémon.
usuarioController.js Cuida dos favoritos e da criação/consulta de equipes.
seed.js Prepara dados iniciais e cria relacionamentos no banco.

A fala principal deve ser: "O backend recebe as ações do frontend, valida os dados, acessa o Neo4j e cria ou consulta os nós e relacionamentos."
