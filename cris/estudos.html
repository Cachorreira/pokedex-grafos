Foco: explicar e demonstrar o ambiente Docker funcionando

1. Ideia central da apresentação

A apresentação deve mostrar que a aplicação foi empacotada em containers Docker e que os serviços conseguem funcionar juntos em um ambiente isolado. O sistema é uma Pokedex em Grafos: o backend Node.js/Express se conecta ao banco Neo4j, grava os dados como nós e relacionamentos, e entrega o frontend já compilado no navegador.

Frase guia para explicar:

"Nosso projeto usa Docker para subir a aplicação e o banco de dados juntos. O container da aplicação roda o backend Node.js e serve o frontend. O container do Neo4j armazena os dados em grafo, com volume persistente para não perder os registros quando o container reinicia."

Parte 2 - Gustavo: Dockerfile linha por linha

Objetivo: explicar como a imagem da aplicação é criada.

FROM node:20-alpine AS frontend-build
WORKDIR /app/frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run build

FROM node:20-alpine AS app
WORKDIR /app
ENV NODE_ENV=production
COPY backend/package*.json ./backend/
RUN cd backend && npm ci --omit=dev
COPY backend ./backend
COPY --from=frontend-build /app/frontend/dist ./frontend/dist
WORKDIR /app/backend
EXPOSE 3000
CMD ["npm", "start"]
Linha Explicação
FROM node:20-alpine AS frontend-build Inicia a primeira etapa da imagem usando Node.js 20 em uma versão leve do Linux. Essa etapa existe para compilar o frontend.
WORKDIR /app/frontend Define a pasta de trabalho dentro do container para os arquivos do frontend.
COPY frontend/package*.json ./ Copia apenas os arquivos de dependências primeiro. Isso melhora o cache do Docker.
RUN npm ci Instala as dependências do frontend de forma reproduzível.
COPY frontend/ ./ Copia o código do frontend para dentro da imagem.
RUN npm run build Gera os arquivos finais do frontend, criando a pasta dist.
FROM node:20-alpine AS app Inicia a segunda etapa da imagem, que será a imagem final da aplicação.
ENV NODE_ENV=production Define o ambiente como produção para o Node.js.
RUN cd backend && npm ci --omit=dev Instala somente as dependências necessárias para rodar o backend.
COPY --from=frontend-build ... Copia o frontend já compilado da primeira etapa para a imagem final.
EXPOSE 3000 Documenta que a aplicação escuta na porta 3000.
CMD ["npm", "start"] Comando executado quando o container da aplicação inicia.
Ponto importante:

O Dockerfile usa multi-stage build. Isso significa que ele compila o frontend em uma etapa e entrega somente o necessário na imagem final, junto com o backend.

Parte 3 - Pedro: Docker Compose, serviços e variáveis

Objetivo: explicar os dois serviços principais: app e neo4j.

services:
  neo4j:
    image: neo4j:5-community
    container_name: pokedex-neo4j
    environment:
      NEO4J_AUTH: neo4j/pokedex123
    ports:
      - "7474:7474"
      - "7687:7687"
    volumes:
      - neo4j_data:/data
      - neo4j_logs:/logs
    networks:
      - pokedex_net

  app:
    build: .
    container_name: pokedex-app
    depends_on:
      - neo4j
    restart: on-failure
    environment:
      PORT: 3000
      NEO4J_URI: bolt://neo4j:7687
      NEO4J_USER: neo4j
      NEO4J_PASSWORD: pokedex123
      JWT_SECRET: pokedex_backend_dev_secret_2026
    ports:
      - "3000:3000"
    networks:
      - pokedex_net
  • neo4j: serviço do banco de dados em grafo.
  • app: serviço da aplicação, contendo backend Node.js e frontend compilado.
  • depends_on: indica que o app depende do Neo4j.
  • restart: on-failure: se o app tentar conectar antes do banco ficar pronto, ele pode reiniciar e tentar novamente.
  • NEO4J_URI: bolt://neo4j:7687: o backend acessa o banco pelo nome do serviço, não por localhost.
Fala sugerida:

"No Docker Compose, os serviços conseguem se enxergar pelo nome. Por isso o backend usa bolt://neo4j:7687. Dentro da rede Docker, neo4j é o endereço do container do banco."

Parte 4 - Matheus: comandos, redes, volumes e persistência

Objetivo: demonstrar os testes básicos do Docker e explicar por que rede e volume são obrigatórios.

Comando O que prova
docker run --rm hello-world Prova que o Docker está instalado e consegue baixar/executar uma imagem.
docker compose up --build Constrói a imagem da aplicação e sobe app + banco juntos.
docker ps Lista os containers em execução.
docker images Mostra as imagens existentes, incluindo pokedex-app.
docker stop teste-stop-rm Para um container de teste.
docker rm teste-stop-rm Remove o container parado.
docker network ls --filter name=pokedex Mostra a rede personalizada pokedex_pokedex_net.
docker volume ls --filter name=pokedex Mostra os volumes pokedex_neo4j_data e pokedex_neo4j_logs.

A rede personalizada permite que o container pokedex-app converse com o container pokedex-neo4j. O volume neo4j_data guarda os dados do banco fora do ciclo de vida do container. Assim, se o container do Neo4j for recriado, os dados podem continuar salvos.

Parte 5 - Peterson: backend, seed, controllers e armazenamento em grafo

Objetivo: explicar como o código usa o banco dentro do ambiente Docker.

  • backend/src/database.js: cria a conexão com o Neo4j usando as variáveis de ambiente.
  • backend/src/seed.js: popula o banco com usuário admin, Pokémon, tipos, habilidades, fraquezas e relações.
  • backend/src/controllers/authController.js: faz cadastro/login, gera JWT e usa bcrypt para senha.
  • backend/src/controllers/pokemonController.js: cadastra e consulta Pokémon e seus relacionamentos.
  • backend/src/controllers/usuarioController.js: gerencia favoritos e equipes do usuário.
  • backend/src/controllers/graphController.js: expõe dados do grafo para visualização.
const driver = neo4j.driver(
  process.env.NEO4J_URI,
  neo4j.auth.basic(process.env.NEO4J_USER, process.env.NEO4J_PASSWORD)
);

Esse trecho representa a conexão do backend com o banco. No Compose, NEO4J_URI recebe bolt://neo4j:7687, então o backend se conecta ao serviço neo4j dentro da rede Docker. 

(u:Usuario)-[:FAVORITOU]->(p:Pokemon)
(u:Usuario)-[:CRIOU_EQUIPE]->(e:Equipe)
(e:Equipe)-[:TEM_MEMBRO]->(p:Pokemon)
(p:Pokemon)-[:TEM_TIPO]->(t:Tipo)
(p:Pokemon)-[:EVOLUI_PARA]->(p2:Pokemon)

Os dados não ficam em tabelas tradicionais. No Neo4j, eles ficam como nós e relacionamentos. Por exemplo, um usuário pode favoritar um Pokémon por meio da relação FAVORITOU, e uma equipe pode conter vários Pokémon por meio da relação TEM_MEMBRO.

6. Como explicar backend, database, seed e controllers

O backend é importante na apresentação, mas deve ser explicado como parte do container app. Ele é o serviço que recebe as requisições do navegador, valida dados, conversa com o Neo4j e devolve respostas em JSON.

8. Perguntas que podem aparecer

Por que o backend usa neo4j no endereço e não localhost?

Porque dentro da rede Docker os containers se comunicam pelo nome do serviço. O nome do serviço do banco no Compose é neo4j.

O que acontece se o container do banco for removido?

Se o volume for mantido, os dados persistem. O volume neo4j_data guarda os arquivos do banco fora do container.

Por que existem duas etapas no Dockerfile?

Para compilar o frontend separado e copiar somente o resultado final para a imagem da aplicação, deixando a entrega mais organizada.

O que o seed faz?

Ele prepara o banco com dados iniciais, como usuário admin, Pokémon, tipos, habilidades e relacionamentos do grafo.
