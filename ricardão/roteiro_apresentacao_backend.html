1. Como organizar a apresentação

A apresentação deve seguir uma sequência lógica: primeiro explicar o que o sistema faz, depois mostrar os recursos da aplicação, em seguida explicar a API e o CRUD, depois a autenticação e segurança, e por fim demonstrar o front-end consumindo a API e o banco Neo4j persistindo os dados.

Ideia central para falar:

"O nosso sistema é uma Pokedex em Grafos. Ele tem uma API REST em Node.js com Express, um front-end em HTML, CSS e JavaScript puro, autenticação com JWT, senhas protegidas com bcrypt e persistência em banco Neo4j. O diferencial é que os dados são armazenados como nós e relacionamentos, o que combina bem com Pokémon, tipos, evoluções, favoritos e equipes."

3. Parte 1 - Visão geral do sistema e recursos

3.1 O que o sistema faz

O sistema permite visualizar Pokémon, consultar suas relações e interagir como treinador. O usuário pode fazer login, cadastrar Pokémon, favoritar Pokémon e criar equipes. A aplicação também exibe um "mapa do grafo", que mostra as relações cadastradas.

3.2 Recursos principais

Recurso Função no sistema Como aparece no banco
Usuario Representa o treinador. Pode se cadastrar, fazer login, favoritar Pokémon e criar equipes. Nó (u:Usuario) com id, nome, email e senhaHash.
Pokemon Recurso principal da Pokedex. Tem número, nome, descrição, imagem, tipo, fraquezas, habilidades e evolução. Nó (p:Pokemon).
Tipo Classifica Pokémon, como Fogo, Água, Grama e Elétrico. Nó (t:Tipo).
Habilidade Descreve habilidades relacionadas a um Pokémon. Nó (h:Habilidade).
Equipe Agrupa Pokémon escolhidos por um usuário. Nó (e:Equipe).
Favoritos Não é uma tabela separada; é uma relação entre usuário e Pokémon. Relação (Usuario)-[:FAVORITOU]->(Pokemon).

3.3 Como explicar os relacionamentos

(Usuario)-[:FAVORITOU]->(Pokemon)
(Usuario)-[:CRIOU_EQUIPE]->(Equipe)
(Equipe)-[:TEM_MEMBRO]->(Pokemon)
(Pokemon)-[:TEM_TIPO]->(Tipo)
(Pokemon)-[:TEM_FRAQUEZA]->(Tipo)
(Pokemon)-[:TEM_HABILIDADE]->(Habilidade)
(Pokemon)-[:EVOLUI_PARA]->(Pokemon)

Em uma apresentação, diga que o Neo4j trabalha com nós e relações. O nó guarda os dados da entidade, e a relação mostra como uma entidade se conecta com outra. Isso é útil porque Pokémon naturalmente possuem ligações: eles têm tipos, fraquezas, evoluções e podem pertencer a equipes.

4. Parte 2 - API REST, rotas e CRUD

4.1 O que é a API no projeto

A API é a camada que recebe as requisições do front-end, conversa com o banco Neo4j e devolve respostas em JSON. Ela foi feita com Node.js e Express. As rotas ficam em backend/src/routes e a lógica fica em backend/src/controllers.

4.2 Exemplo de rotas de Pokémon

router.get('/', ctrl.listar);
router.get('/:id', ctrl.buscar);
router.post('/', auth, validarPokemon, ctrl.criar);
router.put('/:id', auth, validarPokemon, ctrl.atualizar);
router.delete('/:id', auth, ctrl.remover);

Esse trecho mostra o CRUD completo de Pokémon. O GET lista ou busca, o POST cria, o PUT atualiza e o DELETE remove. As rotas que modificam dados usam auth, ou seja, exigem usuário logado.

4.3 Como o CRUD funciona na prática

Operação Rota Explicação
Create POST /pokemons Recebe dados do formulário, valida entrada, cria ou atualiza o nó Pokémon e cria relações com Tipo, Fraqueza, Habilidade e Evolução.
Read GET /pokemons e GET /pokemons/:id Busca Pokémon no Neo4j e devolve JSON para o front-end montar o catálogo.
Update PUT /pokemons/:id Atualiza nome, número, descrição, sprite e substitui relações antigas por novas relações.
Delete DELETE /pokemons/:id Remove o Pokémon e suas relações usando DETACH DELETE.

4.4 Exemplo de leitura do Pokémon com relações

RETURN p {
  .*,
  tipo: head([(p)-[:TEM_TIPO]->(t:Tipo) | t {.*}]),
  fraquezas: [(p)-[:TEM_FRAQUEZA]->(f:Tipo) | f {.*}],
  habilidades: [(p)-[:TEM_HABILIDADE]->(h:Habilidade) | h {.*}],
  evoluiPara: head([(p)-[:EVOLUI_PARA]->(e:Pokemon) | e {.*}])
} AS pokemon

Esse trecho é importante porque mostra que a API não devolve apenas os dados simples do Pokémon. Ela também monta o objeto com os dados relacionados, como tipo, fraquezas, habilidades e evolução.

4.5 CRUD de Tipo

O recurso Tipo também possui CRUD completo. Ele pode ser listado, buscado, criado, atualizado e removido. Isso atende ao requisito de pelo menos dois recursos com CRUD completo. 

router.get('/', ctrl.listar);
router.get('/:id', ctrl.buscar);
router.post('/', auth, validarTipo, ctrl.criar);
router.put('/:id', auth, validarTipo, ctrl.atualizar);
router.delete('/:id', auth, ctrl.remover);

5. Parte 3 - Autenticação, JWT, bcrypt e proteção de rotas

5.1 Cadastro de usuário e bcrypt

No cadastro, a senha nunca é salva em texto puro. Antes de gravar no Neo4j, o sistema usa bcrypt para gerar um hash da senha. 

const senhaHash = await bcrypt.hash(senha, 10);

CREATE (u:Usuario {
  id: randomUUID(),
  nome: $nome,
  email: $email,
  senhaHash: $senhaHash,
  createdAt: datetime()
})

Na fala, explique assim: "O bcrypt transforma a senha em um hash. Mesmo que alguém veja o banco, não encontra a senha original do usuário. O número 10 representa o custo de processamento do hash."

5.2 Login com comparação de senha

const senhaCorreta = await bcrypt.compare(senha, usuario.senhaHash || '');
if (!senhaCorreta) {
  return res.status(401).json({ erro: 'E-mail ou senha invalidos.' });
}

No login, o sistema não descriptografa a senha. Ele compara a senha digitada com o hash salvo. Se não bater, retorna erro 401.

5.3 Geração do token JWT

const token = jwt.sign(
  { id: usuario.id, email: usuario.email },
  process.env.JWT_SECRET,
  { expiresIn: '8h' }
);

Depois do login correto, a API gera um token JWT. Esse token funciona como uma credencial temporária. Ele guarda informações básicas, como id e email do usuário, é assinado com JWT_SECRET e expira em 8 horas.

5.4 Proteção das rotas

if (!authHeader || !authHeader.startsWith('Bearer ')) {
  return res.status(401).json({ erro: 'Token não fornecido.' });
}

const token = authHeader.split(' ')[1];
const payload = jwt.verify(token, process.env.JWT_SECRET);
req.usuarioId = payload.id;
next();

As rotas protegidas usam o middleware auth. Ele verifica se a requisição tem o cabeçalho Authorization: Bearer token. Se o token for válido, o id do usuário fica disponível em req.usuarioId. Se for inválido, a API retorna erro.

5.5 Exemplos de rotas protegidas

router.post('/', auth, validarPokemon, ctrl.criar);
router.put('/:id', auth, validarPokemon, ctrl.atualizar);
router.delete('/:id', auth, ctrl.remover);

router.get('/favoritos', auth, ctrl.favoritos);
router.post('/equipes', auth, ctrl.criarEquipe);

Explique que qualquer pessoa pode visualizar a Pokedex, mas apenas usuários logados conseguem alterar dados, favoritar Pokémon e criar equipes.

6. Parte 4 - Validação, logs e tratamento de erros

6.1 Validação de dados de entrada

Antes de chegar ao controller, os dados passam por middlewares de validação. Isso impede que a API tente salvar registros incompletos ou inválidos. 

function validarPokemon(req, res, next) {
  const { nome, numero, tipoId } = req.body;
  const erros = [];

  if (!nome || nome.trim() === '') erros.push('Campo "nome" e obrigatorio.');
  if (!numero || Number.isNaN(Number(numero))) erros.push('Campo "numero" deve ser numerico.');
  if (!tipoId) erros.push('Campo "tipoId" e obrigatorio.');

  if (erros.length) return res.status(400).json({ erros });
  next();
}

Se houver erro, a API responde com status 400 e uma lista de mensagens. Se estiver tudo certo, chama next() e deixa a requisição continuar.

6.2 Validação de usuário e tipo

if (!email || email.trim() === '') erros.push('Campo "email" e obrigatorio.');
if (!senha || senha.length < 6) erros.push('Campo "senha" deve ter pelo menos 6 caracteres.');

if (cor && !/^#[0-9a-fA-F]{6}$/.test(cor)) {
  erros.push('Campo "cor" deve estar no formato hexadecimal, como #78c850.');
}

No usuário, o sistema exige email e senha com pelo menos 6 caracteres. No tipo, a cor precisa seguir formato hexadecimal, como #78c850.

6.3 Middleware de log

res.on('finish', () => {
  const duracao = Date.now() - inicio;
  const agora = new Date().toISOString();
  console.log(`[${agora}] ${req.method} ${req.originalUrl} → ${res.statusCode} (${duracao}ms)`);
});

Esse middleware registra cada requisição no terminal. Ele mostra método, URL, status da resposta e tempo de execução. Isso ajuda a acompanhar o que está acontecendo no servidor durante testes e apresentação.

6.4 Tratamento centralizado de erros

function erroMiddleware(err, req, res, next) {
  console.error(`[ERRO] ${err.message}`);

  const status = err.status || 500;
  const mensagem = err.message || 'Erro interno do servidor.';

  res.status(status).json({ erro: mensagem });
}

Esse middleware é registrado por último no app.js. Quando algum controller chama next(err), o erro cai nesse ponto central. Assim, a API mantém um padrão de resposta: { erro: "mensagem" }.

7. Parte 5 - Neo4j, persistência e consumo da API pelo front-end

7.1 Conexão com o Neo4j

const driver = neo4j.driver(
  process.env.NEO4J_URI || 'bolt://localhost:7687',
  neo4j.auth.basic(
    process.env.NEO4J_USER || 'neo4j',
    process.env.NEO4J_PASSWORD || 'pokedex123'
  )
);

O arquivo database.js cria a conexão com o banco usando variáveis de ambiente. Em Docker, a URI é bolt://neo4j:7687. Localmente, há um valor padrão para bolt://localhost:7687.

7.2 Funções de leitura e escrita

async function runRead(cypher, params = {}) {
  const session = driver.session();
  try {
    const result = await session.executeRead((tx) => tx.run(cypher, params));
    return result.records;
  } finally {
    await session.close();
  }
}

async function runWrite(cypher, params = {}) {
  const session = driver.session();
  try {
    const result = await session.executeWrite((tx) => tx.run(cypher, params));
    return result.records;
  } finally {
    await session.close();
  }
}

A aplicação separa leituras e escritas. runRead é usado para consultas, e runWrite para criar, atualizar ou remover dados. Em ambos os casos, a sessão é fechada no finally, evitando conexões abertas desnecessárias.

7.3 Persistência dos dados no Neo4j

MATCH (u:Usuario {id: $usuarioId}), (p:Pokemon {id: $pokemonId})
MERGE (u)-[:FAVORITOU]->(p)
RETURN p

Esse trecho cria a relação de favorito. Se a relação já existir, MERGE evita duplicação. Isso é uma vantagem do grafo: o favorito não precisa ser uma tabela intermediária; ele é uma relação entre dois nós.

7.4 Criação de equipes

MATCH (u:Usuario {id: $usuarioId})
CREATE (e:Equipe {id: randomUUID(), nome: $nome, createdAt: datetime()})
MERGE (u)-[:CRIOU_EQUIPE]->(e)
WITH e
UNWIND $pokemonIds AS pokemonId
MATCH (p:Pokemon {id: pokemonId})
MERGE (e)-[:TEM_MEMBRO]->(p)

Quando o usuário cria uma equipe, o sistema cria um nó Equipe, liga essa equipe ao usuário e depois cria relações entre a equipe e cada Pokémon selecionado. O UNWIND percorre a lista de ids enviada pelo front-end.

7.5 Como o front-end consome a API com fetch

async function api(path, options = {}) {
  const headers = { 'Content-Type': 'application/json', ...(options.headers || {}) };
  if (state.token) headers.Authorization = `Bearer ${state.token}`;

  const response = await fetch(`${API_URL}${path}`, { ...options, headers });
  const payload = await response.json().catch(() => ({}));

  if (!response.ok) {
    const details = payload.erro || payload.erros?.join(' ') || 'Falha na requisicao.';
    throw new Error(details);
  }

  return payload;
}

Essa função centraliza as chamadas HTTP do front-end. Ela monta os cabeçalhos, envia o token quando existe, lê a resposta JSON e transforma erros da API em mensagens para a tela.

7.6 Armazenamento do token JWT no front-end

token: localStorage.getItem('pokedexToken') || '',
user: JSON.parse(localStorage.getItem('pokedexUser') || 'null'),

localStorage.setItem('pokedexToken', state.token);
localStorage.setItem('pokedexUser', JSON.stringify(state.user));

Depois do login, o token fica salvo no localStorage. Assim, se o usuário atualizar a página, o front-end consegue recuperar o token e manter o estado de login.

7.7 Envio do token nas requisições protegidas

if (state.token) {
  headers.Authorization = `Bearer ${state.token}`;
}

Sempre que existe token, ele é enviado no cabeçalho Authorization. É isso que permite acessar rotas protegidas, como criar Pokémon, favoritar e criar equipes.

7.8 Quando o token expira ou é inválido

catch (error) {
  state.token = '';
  state.user = null;
  state.favoritos = [];
  state.equipes = [];
  localStorage.removeItem('pokedexToken');
  localStorage.removeItem('pokedexUser');
  setMessage('Sua sessao expirou. Faca login novamente para ver favoritos e equipes.', 'error');
}

Quando a API rejeita o token, o front-end limpa o token salvo, remove o usuário da sessão e mostra uma mensagem pedindo novo login. Esse comportamento atende ao requisito de tratar token expirado ou inválido.
