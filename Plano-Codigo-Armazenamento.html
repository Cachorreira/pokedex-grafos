Pokedex em Grafos

Explicação da estrutura do projeto, papel de cada módulo e forma de armazenamento dos dados: login, Pokémon favoritos e equipes no Neo4j.

1. Visão geral da arquitetura

O projeto foi organizado em três partes: frontend, backend e banco de dados. O frontend é a interface usada pelo usuário. O backend é a API responsável pelas regras de negócio e pela comunicação com o banco. O banco é o Neo4j, que armazena os dados como nós e relacionamentos. 

Frontend HTML/CSS/JavaScript
        |
        | requisições HTTP fetch()
        v
Backend Node.js + Express
        |
        | neo4j-driver + Cypher
        v
Banco Neo4j

No Docker, o projeto é dividido em dois containers principais: pokedex-app, que roda a aplicação, e pokedex-neo4j, que roda o banco.

2. Observação sobre “classes” no projeto

O sistema não usa classes JavaScript no formato class Nome {}. Ele usa módulos CommonJS, com arquivos que exportam funções. Por isso, a melhor forma de explicar o código é por responsabilidade de cada módulo: controllers, rotas, middlewares, banco de dados, seed e frontend.

4. Como os dados são armazenados no Neo4j

No Neo4j, os dados não são organizados em tabelas. Eles são organizados em nós e relações. Isso é adequado para uma Pokédex porque o domínio possui muitas conexões naturais.

DadoForma no Neo4jExemplo
Usuário(u:Usuario)nome, email, senhaHash
Pokémon(p:Pokemon)nome, numero, sprite_url, descricao
Tipo(t:Tipo)Grama, Fogo, Agua, Eletrico
Habilidade(h:Habilidade)Blaze, Torrent, Static
Equipe(e:Equipe)nome da equipe e data de criação

Relacionamentos principais:

(Pokemon)-[:TEM_TIPO]->(Tipo)
(Pokemon)-[:TEM_FRAQUEZA]->(Tipo)
(Pokemon)-[:TEM_HABILIDADE]->(Habilidade)
(Pokemon)-[:EVOLUI_PARA]->(Pokemon)
(Usuario)-[:FAVORITOU]->(Pokemon)
(Usuario)-[:CRIOU_EQUIPE]->(Equipe)
(Equipe)-[:TEM_MEMBRO]->(Pokemon)
A persistência do Neo4j no Docker é feita pelos volumes neo4j_data e neo4j_logs. Mesmo que o container reinicie, os dados continuam salvos no volume.

5. Armazenamento do login

O login possui duas partes: armazenamento no banco e armazenamento temporário no navegador. No banco, o usuário é salvo como nó Usuario. A senha nunca é salva em texto puro; ela é transformada em hash pelo bcrypt e salva como senhaHash. 

const senhaHash = await bcrypt.hash(senha, 10);

CREATE (u:Usuario {
  id: randomUUID(),
  nome: $nome,
  email: $email,
  senhaHash: $senhaHash,
  createdAt: datetime()
})

No login, o backend procura o usuário pelo e-mail e compara a senha digitada com o hash salvo: 

const records = await runRead(
  'MATCH (u:Usuario {email: $email}) RETURN u LIMIT 1',
  { email }
);

const senhaCorreta = await bcrypt.compare(senha, usuario.senhaHash || '');

Se a senha estiver correta, o backend gera um token JWT: 

const token = jwt.sign(
  { id: usuario.id, email: usuario.email },
  process.env.JWT_SECRET,
  { expiresIn: '8h' }
);

O frontend salva esse token no navegador para usar nas próximas requisições protegidas: 

localStorage.setItem('pokedexToken', state.token);
localStorage.setItem('pokedexUser', JSON.stringify(state.user));
O localStorage não é o banco principal. Ele guarda apenas o token e o usuário logado para a sessão do navegador. Os dados reais ficam no Neo4j.

6. Armazenamento dos Pokémon favoritos

Um favorito não é salvo como uma lista dentro do usuário. No Neo4j, ele é salvo como uma relação entre o nó do usuário e o nó do Pokémon. 

MATCH (u:Usuario {id: $usuarioId}), (p:Pokemon {id: $pokemonId})
MERGE (u)-[:FAVORITOU]->(p)
RETURN p

Isso significa que, se o usuário favoritar Pikachu, o grafo fica assim: 

(Usuario {email:'admin@pokedex.local'})
        -[:FAVORITOU]->
(Pokemon {nome:'Pikachu'})

Para consultar os favoritos, o backend percorre essa relação: 

MATCH (u:Usuario {id: $usuarioId})-[:FAVORITOU]->(p:Pokemon)
RETURN p ORDER BY p.numero

A vantagem é que cada favorito é uma conexão navegável. O Neo4j consegue responder naturalmente perguntas como “quais Pokémon esse usuário favoritou?” ou “quais usuários favoritaram este Pokémon?”.

7. Armazenamento das equipes

As equipes são armazenadas como nós próprios, com o rótulo Equipe. O usuário cria uma equipe por meio da relação CRIOU_EQUIPE. Os membros da equipe são ligados pelo relacionamento TEM_MEMBRO. 

MATCH (u:Usuario {id: $usuarioId})
CREATE (e:Equipe {id: randomUUID(), nome: $nome, createdAt: datetime()})
MERGE (u)-[:CRIOU_EQUIPE]->(e)

Depois, para cada Pokémon selecionado, o backend cria a relação de membro: 

UNWIND $pokemonIds AS pokemonId
MATCH (p:Pokemon {id: pokemonId})
MERGE (e)-[:TEM_MEMBRO]->(p)

O grafo resultante fica assim:

(Usuario)-[:CRIOU_EQUIPE]->(Equipe {nome:'Time Kanto'})
                              |
                              +--[:TEM_MEMBRO]->(Pokemon {nome:'Pikachu'})
                              +--[:TEM_MEMBRO]->(Pokemon {nome:'Squirtle'})

Para consultar equipes, o backend busca as equipes criadas pelo usuário e coleta os Pokémon conectados: 

MATCH (u:Usuario {id: $usuarioId})-[:CRIOU_EQUIPE]->(e:Equipe)
OPTIONAL MATCH (e)-[:TEM_MEMBRO]->(p:Pokemon)
WITH e, collect(p {.*}) AS pokemons
RETURN e {.*, pokemons: pokemons} AS equipe
ORDER BY equipe.nome
A equipe pode existir como um nó mesmo se não tiver membros. Nesse caso, ela aparece com pokemons: [].

8. Cadastro e armazenamento dos Pokémon

Ao cadastrar um Pokémon, o backend cria ou atualiza um nó Pokemon usando o número da Pokédex como referência. Em seguida, remove relações antigas de tipo, fraqueza e evolução quando necessário, e cria as relações novas. 

MERGE (p:Pokemon {numero: $numero})
ON CREATE SET p.id = randomUUID(), p.createdAt = datetime()
SET p.nome = $nome,
    p.sprite_url = $sprite_url,
    p.descricao = $descricao,
    p.updatedAt = datetime()

Relação com tipo:

MATCH (tipo:Tipo {id: $tipoId})
MERGE (p)-[:TEM_TIPO]->(tipo)

Relações de fraqueza:

FOREACH (fraquezaId IN $fraquezaIds |
  MERGE (fraqueza:Tipo {id: fraquezaId})
  MERGE (p)-[:TEM_FRAQUEZA]->(fraqueza)
)

Relações de habilidade:

FOREACH (habilidade IN $habilidades |
  MERGE (h:Habilidade {nome: habilidade.nome})
  ON CREATE SET h.id = randomUUID()
  SET h.descricao = habilidade.descricao
  MERGE (p)-[:TEM_HABILIDADE]->(h)
)

10. Frontend e estado da tela

O frontend mantém um estado temporário em memória para renderizar a tela. Esse estado é preenchido com dados vindos da API. 

const state = {
  token: localStorage.getItem('pokedexToken') || '',
  user: JSON.parse(localStorage.getItem('pokedexUser') || 'null'),
  pokemons: [],
  tipos: [],
  graph: { nodes: [], edges: [] },
  favoritos: [],
  equipes: [],
  selectedId: null,
};

Quando a tela carrega, o frontend chama /tipos, /pokemons e /graph. Se houver token, também chama /usuarios/favoritos e /usuarios/equipes.

11. Resumo final

O sistema usa o Neo4j como fonte de verdade. O login fica armazenado como nó Usuario com senha criptografada. Favoritos são relações FAVORITOU. Equipes são nós Equipe conectados ao usuário por CRIOU_EQUIPE e aos Pokémon por TEM_MEMBRO.

Essa modelagem permite que a Pokédex funcione como um grafo real, mostrando conexões entre entidades em vez de guardar tudo em listas isoladas.
